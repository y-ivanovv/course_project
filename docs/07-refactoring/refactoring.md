# Этап 7. Рефакторинг и качество

Рефакторинг проводился отдельной веткой `refactor` и влит в `master`. Все изменения покрыты
тестами (юнит-тесты остаются зелёными).

## 1. Выполненные рефакторинги

| Область | Было | Стало |
|---------|------|-------|
| Пароли | Открытый текст, сравнение строк | BCrypt-хеш, `passwordEncoder.matches` |
| Утечка данных | Контроллеры возвращали `User` с хешем | DTO `UserResponse` без хеша |
| Авторизация | Дублирующиеся `if session==null` в каждом методе | Централизованный `AuthInterceptor` + `@RequireRole` |
| Обработка ошибок | `try/catch` в контроллерах | `@RestControllerAdvice` + `ErrorResponse` |
| Доменные ошибки | `IllegalArgumentException` везде | `ResourceNotFound`/`Duplicate`/`ForbiddenOperation` → коды 404/409/403 |
| Учёт выдач | Только статус книги | Таблица `loans`, привязка к пользователю |
| CORS | `@CrossOrigin` на каждом контроллере | Централизованно в `WebConfig` |
| Фронтенд-состояние | `localStorage` + `window.location.reload()` | `AuthContext` + `useAuth` |
| Дублирование UI | Инлайн-стили во всех компонентах | Общие стили `styles/ui.js`, переиспользуемые компоненты |

## 2. Обязательные паттерны (по методичке)

| Паттерн | Реализация в проекте |
|---------|----------------------|
| **Data Mapper** | Spring Data репозитории (`JpaRepository`, `ElasticsearchRepository`) отделяют бизнес-логику от доступа к данным: сервисы не пишут SQL/запросы ES вручную, а работают с доменными объектами. |
| **Identity Map** | Контекст персистентности Hibernate (first-level cache): в рамках одной транзакции запрос одной и той же сущности по id возвращает один и тот же объект, исключая дубли в памяти. |
| **Unit of Work** | Управление транзакциями Spring (`@Transactional`) в `LoanServiceImpl`/`UserServiceImpl`: изменения фиксируются атомарно на коммите. |

## 3. Статический анализ

- **Frontend:** ESLint (flat config, плагины react-hooks, react-refresh) — `npm run lint`, **0 ошибок**.
  Исправлены: лишние `import React`, неиспользуемые переменные, нарушения правил хуков
  (`react-hooks/set-state-in-effect`, корректные зависимости эффектов).
- **Backend:** рекомендуется подключить SonarQube/Checkstyle (зафиксировано как направление развития);
  ключевые замечания (хранение паролей, дублирование, обработка ошибок) устранены вручную.

## 4. Принципы SOLID

- **SRP** — каждый сервис/контроллер отвечает за свою зону; кросс-каттинг вынесен в перехватчик/advice.
- **OCP/DIP** — Control зависит от интерфейсов сервисов, а не реализаций; `PasswordEncoder` подменяем.
- **ISP** — узкие интерфейсы сервисов и репозиториев.
