# ADR-001: Хранилища данных — Elasticsearch для поиска, PostgreSQL для пользователей и выдач

- **Статус:** Принято
- **Дата:** 2026-03

## Контекст
Требуется полнотекстовый поиск книг по русскому тексту с учётом морфологии (словоформ) и
устойчивостью к опечаткам. Одновременно нужны транзакционность и целостность для данных о
пользователях и выдачах.

## Рассмотренные варианты
1. **SQL `LIKE '%...%'`** в PostgreSQL — просто, но без морфологии, без ранжирования, полное сканирование.
2. **Полнотекстовый поиск PostgreSQL** (`tsvector`, `pg_trgm`) — поддерживает стемминг, но настройка
   нечёткого поиска и ранжирования сложнее, возможности уступают специализированному движку.
3. **Elasticsearch** — специализированный движок: настраиваемые анализаторы, ранжирование (BM25),
   нечёткий поиск (`fuzziness`) «из коробки».

## Решение
- **Каталог книг и поиск** → Elasticsearch (индекс `books`, анализатор `russian_analyzer`:
  токенизатор + lowercase + русские стоп-слова + стеммер).
- **Пользователи и выдачи** → PostgreSQL (JPA), где важны транзакции и связи.

Применён принцип **polyglot persistence** — «правильное хранилище под свой тип данных».

## Последствия
- (+) Качественный ранжированный нечёткий поиск с учётом морфологии.
- (+) Транзакционная целостность для пользователей и выдач.
- (−) Данные книг живут отдельно от реляционной БД — задача согласованности.
- (−) Рост операционной сложности (ещё один сервис, требователен к памяти) — решается через Docker Compose.
