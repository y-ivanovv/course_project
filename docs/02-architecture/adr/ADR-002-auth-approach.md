# ADR-002: Сессионная аутентификация и перехватчик авторизации

- **Статус:** Принято
- **Дата:** 2026-04

## Контекст
SPA-клиенту нужна аутентификация и разграничение ролей (USER/LIBRARIAN). Ранее проверки
авторизации дублировались в каждом методе контроллера (`if session.getAttribute(...) == null`).

## Рассмотренные варианты
1. **Полный Spring Security** (formLogin/UserDetailsService) — мощно, но меняет контракт логина и
   усложняет интеграцию с уже существующим SPA.
2. **Ручные проверки в контроллерах** — дублирование, легко ошибиться.
3. **Сессии + централизованный `HandlerInterceptor`** — единая точка проверки, минимальные изменения контракта.

## Решение
- Аутентификация по email/паролю в `UserService.authenticate` (BCrypt), результат — в серверной сессии.
- Авторизация вынесена в `AuthInterceptor`: проверяет наличие пользователя в сессии (иначе 401) и
  роль по аннотации `@RequireRole` (иначе 403). Публичные эндпоинты (`/login`, `/register`) исключены.
- Пароли хранятся только как BCrypt-хеш; хеш не попадает в ответы (DTO `UserResponse`).

Web-траектория методички допускает «Сессии или JWT» — выбраны сессии.

## Последствия
- (+) Контроллеры без дублирующих проверок, единая политика доступа.
- (+) Чистое разграничение ролей через аннотацию.
- (−) Состояние на сервере (сессия) — для горизонтального масштабирования потребовался бы общий
  store сессий или переход на JWT (зафиксировано как возможное развитие).
